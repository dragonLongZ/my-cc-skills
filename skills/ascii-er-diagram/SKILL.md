# ASCII ER Diagram 绘制规范

## 输出前 Checklist

在输出最终结果之前，逐项检查：

- [ ] 每张表都标注了**中文表名**和**业务用途**
- [ ] 每个字段都有**中文含义**标注
- [ ] 状态/类型类字段标注了**枚举值说明**（如 `状态 0=未付/1=已付/2=退款`）
- [ ] 每个外键关联都写明了**具体字段名**（格式 `FK 字段名 → 父表.字段名`），不能只写 `1:N`
- [ ] 多态关联列出了**所有可能的目标表**
- [ ] 无外键的表标注了"**独立表**"
- [ ] 无物理 FK 但有业务关联的表注明了**关联方式**
- [ ] 表数量 >10 张时，末尾附加了**外键关系汇总表**
- [ ] 按**功能域分组**，主表在上、从表在下

## 核心原则

**一次性给全所有信息，不要让用户反复提醒补充。** 这是本 skill 最重要的原则。如果用户需要提醒你补充信息，那就是失败。

## 标准表框模板

每张表使用以下框结构：

```
┌──────────────────────────────────────────────────┐
│ table_name ─ 中文表名描述                           │
│  业务用途说明                                      │
├──────────────────────────────────────────────────┤
│ PK pk_field        中文含义                        │
│ FK fk_field        中文含义 → 目标表.目标字段       │
│    normal_field    中文含义 枚举值说明              │
│    del_flag        软删除标志                      │
└──────────┬───────────────────────────────────────┘
           │
           │ FK 外键字段名
           │ 1 对 N
           ▼
```

## 字段标注规则

| 字段类型 | 标注格式 | 示例 |
|---------|---------|------|
| 主键 | `PK 字段名 中文含义` | `PK id 主键ID` |
| 外键 | `FK 字段名 中文含义 → 目标表.目标字段` | `FK user_id 用户ID → cyf_consumer_user.user_id` |
| 状态字段 | `字段名 中文含义 状态 0=xx/1=xx/2=xx` | `status 状态 0=未付/1=已付/2=退款` |
| 类型字段 | `字段名 中文含义 类型 1=xx/2=xx` | `business_type 业务类型 1=月卡/2=免费车` |
| 布尔/开关 | `字段名 中文含义 0=否/1=是` | `is_active 是否激活 0=否/1=是` |
| 软删除 | `del_flag 软删除标志` | `del_flag 软删除标志` |
| 普通字段 | `字段名 中文含义` | `phone 手机号` |

## 关联关系表达

- 连接线上标注三要素：**FK 字段名** + **基数关系（1对N / N对1）** + **箭头方向**
- 多态关联要列出所有目标。示例：

```
┌───────────────┐          ┌───────────────┐
│ consumer_user  │          │  cloud_user   │
└───────┬───────┘          └───────┬───────┘
        │ 1                       │ 1
        │ FK user_id              │ FK user_id
        ▼ N                       ▼ N
┌───────────────────────────────────────┐
│        social_user_bind               │
│  FK user_id → consumer_user.user_id   │
│             或 cloud_user.id          │
└───────────────────────────────────────┘
```

## 布局规则

1. **按功能域分组**：相关业务表归为一组，用"一、二、三"分节
2. **主表在上，从表在下**：从表按外键关系向下展开
3. **多对多中间表**放在两个主表之间
4. 每张表的 FK 指向的父表必须已经出现在上方，避免向下引用

## 关键验证规则（易错点）

### 1. 验证表是否在用

不要仅凭实体类存在就判定表在用。当存在多张功能相似的表时（如 `ParkingOrder` 和 `CyfOrder`），必须搜索核心业务流程（支付回调、MQ 消费者、Dubbo 服务实现等）实际引用的是哪张表。**以实际业务调用为准，而非以文件名或类名判断。**

### 2. 关系基数从业务逻辑判断

不要只看数据库 FK 放在哪张表就标 `1:N`。要问"业务上这条记录在现实中对应几条"。例如：

- 一次入场只对应一次出场 = **1:1**，即使 `out_parking.in_parking_id` 的 FK 结构在数据库层面允许 1:N
- 一条出场记录可对应多个订单（如预付+尾款） = **1:N**

**规则：先理解业务流程，再标注基数，不能把数据库约束当成业务关系。**

### 3. 委托搜索后必须交叉验证

使用 Agent/Explore 搜索后，对关键结论（哪张表是主表、关系基数）做二次验证——用 Grep 搜索核心业务文件中的实际引用。不要将 agent 的初步发现直接作为最终结论。

## 汇总表（表数 >10 时必需）

在 ASCII 图形之后附加：

```
## 外键关系汇总

| 子表 | 外键字段 | → 父表 | 父表字段 |
|------|---------|--------|---------|
| social_user_bind | user_id | consumer_user | user_id |
| social_user_bind | social_user_id | social_user | id |

## 独立表（无外键）

- blacklist - 黑名单车辆表
- social_client - 社交OAuth配置表
```

## 示例：小规模 ER 图

用户说："画出用户表和社交账号关联表的 ER 图"

```
一、用户与社交账号体系

┌──────────────────────────────────┐
│ cyf_consumer_user ─ C端用户账号表 │
├──────────────────────────────────┤
│ PK user_id     用户ID            │
│    phone        手机号            │
│    nickname     昵称              │
│    user_balance 用户余额          │
│    del_flag     软删除标志        │
└────────┬─────────────────────────┘
         │
         │ 1
         │ FK user_id
         ▼ N
┌──────────────────────────────────────────────┐
│ cyf_social_user_bind ─ 用户社交账号绑定表     │
├──────────────────────────────────────────────┤
│ PK id                    主键ID              │
│ FK user_id               用户ID              │
│    → cyf_consumer_user.user_id              │
│ FK social_user_id        社交用户ID          │
│    → cyf_social_user.id                     │
│    social_type           平台类型             │
│    status                状态 0=正常/1=禁用   │
└───────────────────┬──────────────────────────┘
                    │
                    │ FK social_user_id
                    │ 1
                    ▼ N
┌──────────────────────────────────┐
│ cyf_social_user ─ 社交平台用户表  │
├──────────────────────────────────┤
│ PK id            主键ID          │
│    social_type   平台类型         │
│    open_id       平台OpenID      │
│    phone         手机号           │
│    status        状态             │
└──────────────────────────────────┘
```
