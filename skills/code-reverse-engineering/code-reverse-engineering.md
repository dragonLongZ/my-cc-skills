# 代码逆向分析验证流程

## 核心原则

**基于代码事实，而非方法名或业务直觉做假设。**

>  **强制要求**：分析代码时，必须先加载本技能，然后逐条遵守以下规则。违反任何一条都可能导致分析结论完全错误。

## 执行约束（违反后果严重）

1. **禁止基于方法名推断行为** — 必须查看方法内部实现
2. **禁止基于参数名推断含义** — 必须追踪参数来源和传递过程
3. **禁止基于业务直觉做结论** — 必须用代码证据验证
4. **禁止只看方法签名** — 必须阅读方法体内部逻辑
5. **禁止假设 SQL 范围** — 必须查看 WHERE 条件和 JOIN 逻辑

### 历史教训（2026-04-28）

因违反上述规则导致的错误：
- 看到 `getMemberInParkingCount` 就假设"统计同会员所有在停车辆"，实际 SQL 有 `AND charge_rule_id IS NULL` 条件，只统计纯会员车
- 看到 `calTempCarFeeOutProcess(..., outTime)` 就假设 outTime 是"出场时间"，实际它是"上一辆会员车的出场时间"，用于计算计费时段
- 结果：整个计费逻辑分析完全错误，需要推倒重来

## 验证 Checklist

分析任何代码前，必须完成以下检查：

- [ ] **SQL 定义**：查看实际 SQL 查询，确认 WHERE 条件、JOIN 逻辑、统计范围
- [ ] **变量生命周期**：追踪关键变量的来源、传递过程、最终使用位置
- [ ] **方法实现**：阅读方法内部逻辑，不只依赖方法签名
- [ ] **假设验证**：每个结论都有代码证据支撑
- [ ] **完整阅读**：不基于部分信息就发布结论

## 常见陷阱

| 陷阱 | 正确做法 |
|------|----------|
| 看方法名 `getMemberInParkingCount` 假设统计所有车辆 | 查看 SQL：`WHERE charge_rule_id IS NULL`，只统计纯会员车 |
| 看参数 `outTime` 假设是出场时间 | 追踪来源：`outTime = getLastMemberCarOutTime()`，是上一辆会员车出场时间 |
| 看方法签名 `calTempCarFeeOutProcess(..., outTime)` | 阅读实现：计费时段 = `[入场时间 → outTime]`，不是 `[入场→出场]` |

## 分析步骤

1. **定位核心方法**：找到业务流程的入口方法
2. **追踪调用链**：按执行顺序追踪方法调用
3. **检查 SQL**：查看每个查询的实际 WHERE 条件和统计逻辑
4. **追踪变量**：关键变量（如 outTime、memberInParkingCount）的来源和传递
5. **验证假设**：每个结论都有代码证据
6. **完整验证**：覆盖所有分支和边界情况

## 输出要求

- 每种情况都要逐条展开，包含完整的推理过程
- 包含：判定条件、变量值、计费结果、计费时段
- 不能简化为表格中的一行结论
- 标注代码位置和行号

## 反思机制

分析完成后，自问：
1. 我的每个结论都有代码证据吗？
2. 我是否追踪了关键变量的完整生命周期？
3. 我是否基于业务直觉做了未经代码验证的假设？
4. 如果用户质疑，我能指出具体的代码行吗？
